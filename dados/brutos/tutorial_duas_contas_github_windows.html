Git + duas contas do GitHub no Windows

Tutorial revisado com os erros reais que apareceram no seu caso, indicação clara de onde cada etapa deve ser feita e uma navegação lateral.
Ideia central
O mais seguro é organizar por projeto/repositório, não por aplicativo. Você pode abrir um projeto no RStudio ou no Positron sem conflito, desde que o repositório esteja com o remote, o e-mail e a chave SSH corretos.
Contas usadas no exemplo
Conta pessoal → alias github-pessoal
Conta institucional → alias github-inst

1) Instalação inicial

Etapa 1

Fazer no Git Bash

Se ainda não tiver o Git instalado, instale o Git for Windows. Depois confirme que Git e SSH estão disponíveis:

git --version
ssh -V

Se esses dois comandos funcionarem, você pode continuar.

2) Criar duas chaves SSH

Etapa 2

Fazer no Git Bash

Crie uma chave para cada conta. Não misture as duas.

ssh-keygen -t ed25519 -C "seu-email-pessoal@exemplo.com" -f ~/.ssh/id_ed25519_pessoal

ssh-keygen -t ed25519 -C "seu-email-institucional@exemplo.com" -f ~/.ssh/id_ed25519_inst

Você pode definir passphrase. É mais seguro. Depois vamos ver como não precisar digitá-la toda hora.

3) Conferir se os arquivos foram criados

Etapa 3

Fazer no Git Bash 
ls -la ~/.ssh

Você deverá ver algo parecido com isto:

id_ed25519_pessoal
id_ed25519_pessoal.pub
id_ed25519_inst
id_ed25519_inst.pub

4) Adicionar as chaves ao GitHub

Etapa 4A

Primeiro no Git Bash

Copie a chave pública. No Windows, prefira clip.

clip < ~/.ssh/id_ed25519_pessoal.pub

Depois, para a outra conta:

clip < ~/.ssh/id_ed25519_inst.pub

Se o cat arquivo.pub der erro, use clip ou abra com o Bloco de Notas.

notepad ~/.ssh/id_ed25519_pessoal.pub
notepad ~/.ssh/id_ed25519_inst.pub
Etapa 4B

Depois no GitHub web

Em cada conta do GitHub, vá em:

Settings → SSH and GPG keys → New SSH key
  • Title: por exemplo, Notebook Windows - Pessoal ou Notebook Windows - Institucional
  • Key type: deixe Authentication Key
  • Key: cole o conteúdo da chave .pub
Importante
A chave id_ed25519_pessoal.pub vai para a conta pessoal.
A chave id_ed25519_inst.pub vai para a conta institucional.

5) Criar o arquivo de configuração SSH

Etapa 5

Fazer no Git Bash

Abra o arquivo:

notepad ~/.ssh/config

Cole este conteúdo:

Host github-pessoal
  HostName ssh.github.com
  Port 443
  User git
  IdentityFile ~/.ssh/id_ed25519_pessoal
  IdentitiesOnly yes

Host github-inst
  HostName ssh.github.com
  Port 443
  User git
  IdentityFile ~/.ssh/id_ed25519_inst
  IdentitiesOnly yes
Atenção
Salve o arquivo como config, sem .txt.
Se o Windows salvar como config.txt, o SSH não vai reconhecer os aliases.

No seu caso, a porta 22 estava bloqueada. Por isso usamos ssh.github.com com Port 443.

6) Testar as duas contas

Etapa 6

Fazer no Git Bash 
ssh -T git@github-pessoal
ssh -T git@github-inst

Na primeira conexão, pode aparecer uma mensagem pedindo confirmação do host. Digite:

yes

O resultado esperado é algo como:

Hi astuciasnor! You've successfully authenticated, but GitHub does not provide shell access.
Hi cluberufpa! You've successfully authenticated, but GitHub does not provide shell access.

Isso significa que está funcionando.

7) O que fazer se der erro

ErroO que significaComo corrigir
Permission denied ao ler .pub O arquivo existe, mas o comando usado não ajudou Use clip < arquivo.pub ou notepad arquivo.pub
Could not resolve hostname github-pessoal O alias não foi reconhecido Verifique se o arquivo é ~/.ssh/config e não config.txt
Connection timed out port 22 A porta 22 foi bloqueada na rede Use ssh.github.com com Port 443 no arquivo config
git push pedindo senha do GitHub O repositório provavelmente está em HTTPS, não em SSH Confira git remote -v. O ideal aqui é ver git@github-pessoal:... ou git@github-inst:...

8) Ajustar o remote de cada projeto

Etapa 8

Fazer na pasta do projeto, no Git Bash

Entre no repositório e aponte o origin para a conta certa.

Projeto pessoal

git remote set-url origin git@github-pessoal:astuciasnor/meu-projeto.git

Projeto institucional

git remote set-url origin git@github-inst:cluberufpa/repositorio-institucional.git

Confira:

git remote -v

9) Definir nome e e-mail por repositório

Etapa 9

Fazer na pasta do projeto, no Git Bash

Isso é o que vincula o commit à conta certa.

Projeto pessoal

git config user.name "astuciasnor"
git config user.email "astuciasnor@gmail.com"

Projeto institucional

git config user.name "cluberufpa"
git config user.email "cluber.ufpa@gmail.com"

Para conferir:

git config --get user.name
git config --get user.email

10) Exemplo completo de uso

Exemplo prático

Fazer na pasta do projeto, no Git Bash

Se for abrir no RStudio, abra o arquivo do projeto que fica dentro da pasta, normalmente com extensão .Rproj. O ideal é abrir o projeto correto, e não apenas uma pasta solta.

Projeto pessoal

cd /c/caminho/do/projeto
git remote -v
git remote set-url origin git@github-pessoal:astuciasnor/meu-projeto.git
git config user.name "astuciasnor"
git config user.email "astuciasnor@gmail.com"
git remote -v
git config --get user.email

Projeto institucional

cd /c/caminho/do/projeto
git remote set-url origin git@github-inst:cluberufpa/repositorio-institucional.git
git config user.name "cluberufpa"
git config user.email "cluber.ufpa@gmail.com"

11) RStudio e Positron

OndeO que acontece
RStudioUsa o Git do sistema. Abra o arquivo do projeto dentro da pasta, normalmente o .Rproj. Se o projeto já estiver configurado, o RStudio respeita o remote e o e-mail do repositório.
PositronTambém usa o Git do sistema. Você pode abrir a pasta do projeto normalmente. O importante não é a IDE, e sim a configuração do projeto.
Resumo
Você não precisa “trocar de conta” manualmente ao mudar de IDE. O que define a conta é o próprio repositório.

12) Como parar de digitar senha / passphrase toda vez

Opção recomendada

Fazer no Git Bash

Se você estiver usando SSH, o ideal não é remover a proteção da chave, e sim usar o ssh-agent para lembrar a passphrase durante a sessão.

eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519_pessoal
ssh-add ~/.ssh/id_ed25519_inst

Na primeira vez, ele pedirá a passphrase das chaves. Depois disso, normalmente você não precisará digitá-la a cada operação.

Automatizar no Git Bash

Fazer no Git Bash

Para não repetir isso sempre que abrir o Git Bash, abra:

notepad ~/.bashrc

E adicione este bloco no final do arquivo:

env=~/.ssh/agent.env

agent_load_env () { test -f "$env" && . "$env" >| /dev/null ; }

agent_start () {
    (umask 077; ssh-agent >| "$env")
    . "$env" >| /dev/null
}

agent_load_env

agent_run_state=$(ssh-add -l > /dev/null 2>&1; echo $?)

if [ ! "$SSH_AUTH_SOCK" ] || [ $agent_run_state = 2 ]; then
    agent_start
    ssh-add ~/.ssh/id_ed25519_pessoal
    ssh-add ~/.ssh/id_ed25519_inst
elif [ "$SSH_AUTH_SOCK" ] && [ $agent_run_state = 1 ]; then
    ssh-add ~/.ssh/id_ed25519_pessoal
    ssh-add ~/.ssh/id_ed25519_inst
fi

Depois salve, feche e abra o Git Bash novamente.

Se você quiser remover a passphrase totalmente
Isso é possível, mas é menos seguro. Só faça se realmente quiser abrir mão dessa proteção. 
ssh-keygen -p -f ~/.ssh/id_ed25519_pessoal
ssh-keygen -p -f ~/.ssh/id_ed25519_inst

Quando pedir a nova passphrase, deixe em branco.

13) Checklist rápido para novos projetos

  • Entrar na pasta do projeto
  • Verificar git remote -v
  • Apontar para github-pessoal ou github-inst
  • Definir git config user.email corretamente
  • Se estiver usando RStudio, abrir o arquivo .Rproj do projeto
  • Fazer um teste com git push

14) Comentário final

O principal erro ao usar duas contas GitHub no mesmo computador não é o RStudio nem o Positron. O problema quase sempre está em um destes pontos:

  • remote apontando para a conta errada
  • e-mail do commit errado
  • arquivo config salvo como config.txt
  • porta 22 bloqueada, exigindo uso da porta 443
  • chaves não carregadas no ssh-agent, fazendo você digitar a passphrase toda hora

Quando esses pontos estão corretos, o uso de duas contas fica estável e compatível.
